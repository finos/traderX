The "Trading Application Pet Store"

In software training, a "pet store" is a simplified, manageable application. TraderX mimics the core functionalities of complex trading platforms found at major FSIs, but is built approachably so it can run locally on a laptop.

The Goal

To provide a safe experimentation sandbox. Bridge the theory-practice gap by translating abstract concepts into practical solutions without relying solely on slide decks.

Serving FINOS

Accelerate adoption of FINOS standards by providing a unified, tangible showcase. Evaluators can assess how projects like FDC3 or CDM fit into a relatable architecture.

The Audiences

Serves new developers learning fundamentals, tech teams testing integrations, and business stakeholders needing real-time visual demos of proposed investments.

Eliminating Fork Drift with SDD

To maintain 14+ distinct learning states without creating an unmergeable nightmare of branches, we moved to Spec-Driven Development. The main branch contains NO runtime code—only specifications. The runtime code is deterministically generated into isolated code/generated-state-* branches.

Real-time Push vs. Polling (State 009) Angular / Socket.IO

Instead of HTTP polling, State 009 uses SDD to wire the Order Matcher directly to the PubSub messaging bus. The front-end is spec'd to subscribe dynamically to trades, orders, and positions.

// web-front-end/angular/main/app/trade/order-blotter
// Real-time updates replace the 5000ms polling loop
const unsub = this.orderAdminService.subscribe(
  `/accounts/${accountId}/orders`,
  (order: OrderRecord) => this.applyOrderUpdate(order)
);
this.subscriptions.add({ unsubscribe: unsub });
Environment Contracts TypeScript

SDD enforces structural completeness at compile time by requiring all generated UI states to adhere to an Environment Interface, preventing silent runtime failures.

export interface Environment {
  production: boolean;
  accountUrl: string;
  refrenceDataUrl: string; // Intentional typo matching legacy
  tradesUrl: string;
  positionsUrl: string;
}
Backpatching & Security Audits

We utilize Spec-Driven Development to maintain security baselines across the ecosystem. Tools like Backpatch run through our generated branches, seamlessly upgrading dependencies (e.g., Spring Boot, Netty) in Artifactory while automatically logging justification metadata to prevent scanner false positives.

  • CVE SLAs: Dependencies are strictly tracked in the Constitution matrices.
  • LLM-Native Docs: Documentation is isolated and compiled directly to Docusaurus, generating an LLMs.txt sitemap for AI consumption.
  • Never touch the codebase: PRs strictly target the Spec files. The generators do the rest.

The GitHub Spec Kit Engine

Spec Kit replaces non-deterministic "AI vibe coding" with a highly structured pipeline. AI assists with reasoning and technical decomposition, but deterministic patch scripts control the exact files emitted.

1

speckit.init

Establishes the repository's Constitution. Encodes corporate guardrails (AIGF v2.0), branch protection rules, and dependency matrices (e.g., Backpatch enforcement).

2

speckit.specify

Collaborate with an LLM to translate business directives into a strict requirements.yml, defining API contracts, flows, and testing parameters.

3

speckit.plan & .tasks

The system slices the specification into human-reviewable, atomic engineering tasks to prevent context-destroying mega-commits.

4

speckit.implement

Executes the generation. Target code branches are built deterministically from patch files. CI/CD automatically deploys the containers.

Example: The Constitution YAML 
# .speckit/constitution.yml
governance:
  compliance:
    - rule: "AIGF_v2.0_AIR_OP_005"
      enforcement: strict
      description: "Foundation Model versions must be pinned."
  architecture:
    - rule: "NO_LOCAL_STATE"
      description: "Microservices must use external persistence."
